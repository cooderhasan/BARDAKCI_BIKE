2  Temel Bulgular
Test Summary
The frontend suite is moderately healthy in the executable subset: 11 passed, 3 failed, and 10 blocked. Because
blocked cases are excluded from scoring, the quality score is computed on the passed/failed subset only, which yields
a decent but not strong result. The main stability issues are concentrated in checkout/payment and a few account ﬂows,
while catalog browsing and several customer account actions are working correctly. The blocked tests are mostly due
to missing or rejected admin/customer credentials, so they do not reﬂect product behavior directly, but they do limit
coverage of critical admin workﬂows.
What could be better
The two real regressions are in core commerce ﬂows. First, the site allows adding an out-of-stock item even though
`valuemax=0` is present and the page still shows `Stokta Mevcut`; the cart accepts the item and shows a success toast.
Second, PayTR checkout fails with `Gecersiz istek, post icerigini kontrol edin`, which suggests malformed or invalid
payment-request parameters. There is also an account-registration issue: the customer is created but not signed in
automatically after `Kayıt Başarılı`. Several admin and account tests were blocked by rejected login attempts, which
suggests missing test credentials or an auth configuration problem in the environment.
Recommendations
Fix the two checkout-blocking defects first: enforce server-side stock validation before add-to-cart, and inspect the
PayTR request payload/signature generation end to end so the POST body matches gateway requirements exactly.
Next, update registration so successful signup establishes an authenticated session and redirects into the account area.
In parallel, provision stable admin/customer test accounts or a staging auth bypass so the blocked admin/order-
management tests can run. After each change, re-run the affected ﬂows and verify the exact titles in the suite: out-of-
stock add-to-cart, PayTR card checkout, automatic login after registration, and the admin order/inventory pages.

TEST FAILURE The site allowed adding an item to the cart despite inventory indicators showing zero stock. Observations:
- The quantity input has valuemax=0 indicating no available stock, but the product page also shows a 'Stokta Mevcut'
label. - After clicking 'Sepete Ekle', a success toast 'Ürün sepete eklendi!' appeared and the cart total updated to ₺ 250. -
The add-to-cart button was enabled and addition proceeded instead of being prevented.
Cause
The most likely cause is an inconsistency between the front-end availability indicator and the server-side inventory
validation. The page shows valuemax=0, suggesting no stock, but the product is still marked as 'Stokta Mevcut' and the
cart endpoint accepts the item, which points to stale or mismatched stock data, a caching/synchronization issue, or a
missing server-side check that allows cart addition even when inventory is zero.
Fix
Correct the inventory-to-purchase logic on the hosting/application side so the add-to-cart endpoint and UI both enforce
the same stock source of truth. If the product is truly out of stock, return a non-success response from the server when
stock is 0, disable/lock the add-to-cart action, and update the product label to match the real inventory state. If the
product is meant to be sellable despite the UI showing 0, fix the stock sync so inventory is reported consistently (e.g.,
cache/index/ERP sync, template data, or product variant stock mapping) and ensure valuemax, availability labels, and
cart validation are derived from the same backend value.
